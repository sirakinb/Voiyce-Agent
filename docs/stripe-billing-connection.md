# Stripe Billing Connection

This document explains how Voiyce is currently wired to Stripe.

## Current status

- Local source review now blocks accidental Stripe live-mode use in checkout, billing portal, and billing sync.
- `STRIPE_SECRET_KEY=sk_live_...` is rejected unless `STRIPE_ALLOW_LIVE_MODE=true` is also set.
- For beta, leave `STRIPE_ALLOW_LIVE_MODE` unset and use Stripe test mode.
- A Stripe webhook secret is configured.
- A fallback Stripe price ID is configured.
- Dedicated `STRIPE_MONTHLY_PRICE_ID` and `STRIPE_YEARLY_PRICE_ID` secrets are not currently set.

## Connection path

### 1. The app opens checkout through an InsForge function

The Mac app does not call Stripe directly.

Instead:

- `BillingManager.beginCheckout()` invokes the InsForge edge function `create-checkout-session`
- `BillingManager.openBillingPortal()` invokes the InsForge edge function `create-portal-session`

## 2. The checkout function authenticates the user

`insforge/functions/create-checkout-session/index.ts`:

- reads the current authenticated user from InsForge auth
- finds or creates a `billing_profiles` row
- stores the user's preferred plan
- creates a Stripe customer if one does not already exist

## 3. The checkout function talks to Stripe server-to-server

The same function then sends a server-side request to:

- `https://api.stripe.com/v1/checkout/sessions`

using:

- `STRIPE_SECRET_KEY`

That returns a Stripe Checkout URL, which the app opens in the browser.

Live-mode guardrail:

- `create-checkout-session`, `create-portal-session`, and `sync-billing-status` call the Stripe API only after the secret key passes `requireStripeSecretKey()`.
- Live secret keys require the explicit `STRIPE_ALLOW_LIVE_MODE=true` environment acknowledgment.
- Deno coverage verifies all three paths block `sk_live_...` before any network request runs.

## 4. Stripe sends subscription updates back to Voiyce

Stripe webhook events are handled by:

- `insforge/functions/stripe-webhook/index.ts`

That function:

- verifies the `Stripe-Signature` header using `STRIPE_WEBHOOK_SECRET`
- accepts subscription created, updated, and deleted events
- extracts customer, subscription, price, period end, and plan info
- writes the update into the database through `public.apply_stripe_subscription_update`

## 5. The app refreshes billing access after checkout

After Stripe Checkout closes:

- the browser is sent to the InsForge function `billing-return`
- that returns the user to the app through the billing callback flow
- the app calls `sync-billing-status`
- `BillingManager` refreshes access and removes the paywall if the subscription is active

## Current pricing configuration nuance

The checkout function supports three secret-based price inputs:

- `STRIPE_MONTHLY_PRICE_ID`
- `STRIPE_YEARLY_PRICE_ID`
- `STRIPE_PRICE_ID`

Current linked backend state:

- `STRIPE_PRICE_ID` is present
- `STRIPE_MONTHLY_PRICE_ID` is missing
- `STRIPE_YEARLY_PRICE_ID` is missing

That means:

- monthly checkout can use the fallback `STRIPE_PRICE_ID`
- yearly checkout falls back to inline `price_data` generated by the function

Inline yearly fallback in code:

- amount: `12000`
- currency: `usd`
- interval: `year`

## Important implication

If you want both plans to be managed fully inside Stripe Products/Prices, you should set:

- `STRIPE_MONTHLY_PRICE_ID`
- `STRIPE_YEARLY_PRICE_ID`

That removes the inline yearly fallback and makes both plans explicit Stripe prices.

Before charging real users, also confirm the Stripe account mode, live product IDs, live webhook endpoint, live webhook secret, billing portal cancellation behavior, refund copy, and the exact subscription/cancellation language shown in Voiyce.
